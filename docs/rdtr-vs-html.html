RDTR vs HTML

This document explains how RDTR differs from the traditional web stack.

1. Architectural Philosophy

Concept HTML Stack RDTR
Layout CSS-based, cascading Constraint-based, no cascade
Styling CSS None
Scripting JavaScript Sandboxed WASM
Rendering Browser engine Deterministic runtime
Protocol HTTP (text) Binary over QUIC
Positioning Arbitrary, absolute allowed No absolute positioning

2. Complexity Model

HTML

  • DOM tree
  • CSS cascade
  • Event bubbling
  • Reflow + repaint
  • Implicit layout dependencies

RDTR

  • Layout tree
  • Overlay stack
  • Explicit mutation operations
  • Deterministic resolution
  • No hidden layout state

3. Security Surface

HTML

  • XSS
  • CSP complexity
  • DOM injection
  • CSS side channels

RDTR

  • WASM sandbox
  • Validated mutation model
  • No runtime code injection
  • No styling layer

4. Rendering Guarantees

HTML

  • Layout depends on browser heuristics
  • Rendering differences between engines

RDTR

  • Rendering must be deterministic
  • Compliance test suite enforces correctness

5. Network Model

HTML

  • Text-based
  • Verbose
  • Implicit state

RDTR

  • Compact binary frames
  • Explicit event → mutation cycle
  • Deterministic replay possible

6. Intended Use

HTML

  • General purpose UI system
  • Consumer web

RDTR

  • Secure terminal interfaces
  • Remote-first systems
  • Infrastructure dashboards
  • SSH-native tools

7. Summary

RDTR is not a replacement for the browser.

It is a structured, deterministic runtime for terminal-native environments.

HTML optimizes for flexibility. RDTR optimizes for determinism and safety.
